Agents are good enough to build real applications, but they keep hitting the same walls.

> "I've written the migration. Can you run it? I don't have permission to execute against the database."

> "I've set up the OAuth handlers, but I can't create the GCP project needed for credentials. Can you create a project in the console and add the OAuth redirect URI? I'll need the client ID and secret to continue."

> "I've built the signup flow. Can you test it manually? I'll need a real email and phone number to get through verification."

The agent wrote the code, but it can't provision what it needs, deploy what it built, or verify what it produced. To unlock truly autonomous agents, we believe agents must be free to work however they see fit — which requires a development philosophy that encourages that freedom without sacrificing safety.

## What Blocks Agents

Agents working on real applications hit three categories of wall.

**Access** is the most basic layer. Agents need permission to act on your machine — writing files, running shell commands, installing packages, making network requests. An unrestricted agent can do real damage, so frameworks gate actions behind approval prompts. Even when the agent is doing exactly what you asked, the permission model assumes it might not be.

Beyond local permissions, the agent needs credentials for the services the code depends on — API keys, cloud credentials, deployment tokens, database connection strings. Most teams keep these tightly scoped or out of the agent's reach entirely, which means the agent writes a perfectly good migration and then asks you to run it.

**Infrastructure** is what the application runs on. When a developer joins a team, they get a seat on the org — they can provision what they need, from spinning up a database to configuring a deployment. Agents don't get seats, they get environment variables. They can connect to existing services but can't create new ones. The moment a task requires provisioning, the agent is stuck.

**Identity** is the problem of being a person. Software sends confirmation emails, redirects to consent screens, validates credit cards against billing addresses, delivers SMS codes to phone numbers. Testing these flows requires a real human identity on the other side — an email inbox that receives mail and a phone number that carriers don't flag as VOIP. The agent has no way to test any of this.

Every one of these blockers ends the same way: either the agent stops and asks you for help, or it skips what it can't do and assumes it works.

## Give Them Their Own Org

The instinct is to carefully expand what agents can do inside your existing environment. Grant a few more permissions. Add a scoped credential. Share a staging key. This is the wrong direction. The more access the agent has to your infrastructure, the more you need to supervise it, which defeats the point.

The better answer: don't give agents more access to your environment. Give them their own.

We provisioned an entirely separate organization — a complete fork of our company's project structure under completely different accounts. Separate GitHub org, separate AWS and GCP accounts, separate Vercel workspace, separate billing. No production data. No customer access. Projects get cloned in and agents get full control.

Inside this org, the agent doesn't request access. It has it. Every resource maps to one of the three blockers, and every blocker has a concrete solution.

**Full permissions for access.** Skip confirmation prompts. Approve tool calls. Let the agent write and execute freely.

**Real seats for infrastructure.** The agent gets admin accounts on every platform it needs, not scoped credentials. It can create cloud projects, provision databases, deploy applications, and configure DNS — the way a new developer would on their first week. GitHub with full push access and CI. Sandbox AWS and GCP accounts where the agent creates IAM policies and configures OAuth credentials. Its own Vercel workspace to deploy, set environment variables, and attach domains. Admin access to Neon for Postgres databases and Upstash for Redis instances. Ngrok to expose local services for webhook testing and OAuth callbacks. A real domain with subdomains, DNS records, and HTTPS.

**Real identities for testing.** The agent gets an email address that receives mail, for confirmation links and verification codes. A phone number that passes carrier verification. A card that clears AVS checks with a hard spending cap. Playwright sessions for navigating consent screens, submitting forms, and verifying UI flows. For scarce resources, we maintain pools of pre-provisioned identities that agents can draw from and return.

With this setup, the agent can provision a database, deploy against it, configure OAuth with a real callback URL, sign in with a real account, and verify the entire flow. No yielding. No asking for help.

## The Isolation Boundary

An agent with admin access to cloud accounts, a credit card, and the ability to run arbitrary code is a legitimate concern. This works because none of these resources are connected to production.

There is no IAM role, no shared credential, and no network path between the agent's org and ours. An agent with root access inside its sandbox cannot reach our production infrastructure. The two orgs are completely separate entities with completely separate billing.

<Figure id="one-way-bridge">
  <OneWayBridgeFigure />
  <Figure.Caption>A project forks into the sandbox, gets modified and verified, then returns as a pull request. Nothing else crosses the boundary.</Figure.Caption>
</Figure>

Code flows in one direction. When the agent finishes a task, the result is a pull request into our production org. Our CI runs. Our team reviews. We merge or we don't. The agent proposes changes. We accept or reject them. That's the only bridge between the two worlds.

We run agent workloads on dedicated Mac Minis in isolated VMs. The card has a hard spending cap. Cloud accounts have billing alerts. If an agent provisions something expensive or gets stuck in a loop, the blast radius is a capped card and a sandbox account. Not our infrastructure.

The safety model is not about restricting what the agent can do. It's about ensuring that nothing the agent does can reach production.

## What's Next

This setup gives agents what they need to build real software end to end. Full permissions, real infrastructure, real identity — all inside an environment that can't touch production. The bottleneck is no longer the environment.

But unblocked doesn't mean correct. We've had agents build an app, deploy it, verify their own work, and report success — only to find the software was still wrong. Not because the agent lacked access, but because it didn't know what "correct" meant. Or it knew at the start and drifted. Or it checked its work but checked the wrong things.

That's a specification problem. We wrote about how we're approaching it in [Contract Engineering](/lab/contract-engineering).
